# 群共享空间 · 实现与自测报告

> 形态：纯 P2P 局域网群聊（iLink / 仓库 freedom），Electron 主进程 + React 渲染进程。
> 本轮新增「群共享空间」功能：无中心服务器、无云盘架构、不做编辑锁、不做实时协同、上传替换生成新版本不覆盖。

---

## 1. 功能实现概述

在现有群聊中新增「共享空间」入口。一个群可创建多个共享空间，**谁创建谁的电脑就是该空间的共享主机**，全部文件与元数据存储在主机本机。群内任何人都可向任意空间上传文件/文件夹、新建文件夹、下载、上传新版本、查看历史版本；删除/重命名限主机或群主。主机离线时空间不可访问并明确提示，可展示标注「缓存数据，可能不是最新」的本地快照。

复用现有 P2P 与文件传输能力：

- **目录列表 / 新建文件夹 / 历史 / 重命名 / 删除 / 下载请求**走新增的加密单播控制信令 `share`（请求-响应、reqId 去重）。
- **文件内容**走既有 `FileTransfer` TCP 直连（分片加密、SHA-256、`.part` 暂存、断点续传）。
- **功能广播**（创建/上传/新建文件夹/新版本/重命名/删除）复用既有群聊系统消息可靠投递（在线广播 + 离线发件箱补发），**仅在聊天框展示、不弹通知**（满足「广播只在聊天框展示」要求）。
- 极简展示：列表/目录只显示核心信息，创建者、主机设备、SHA-256、时间等附加信息走鼠标悬浮提示（`title`）。

## 3. 新增数据结构

**vault 顶层（`store.enc` 内，加密持久化）**

- `shareSpaces: SpaceRecord[]`：`{ spaceId, groupId, name, hostUserId, hostDeviceId, hostIp, rootPath, createdBy, createdAt, updatedAt, status, fileCount, isHost }`（`rootPath` 仅宿主本机有意义）。
- `shareSnapshots: { [spaceId]: { [parentId]: { entries, breadcrumb, ts } } }`：目录缓存快照，供主机离线时只读展示。

**宿主磁盘 `<rootPath>/.ilink-share/meta.json`（权威）**

- `space`：空间元信息（含 status normal/deleted）。
- `entries: { [entryId]: Entry }`：目录项。`{ entryId, spaceId, groupId, parentId, name, type(file/folder), ext, logicalBase, currentVersion, currentVersionId, size, hash, relativePath, createdBy, updatedBy, createdAt, updatedAt, status }`。
- `versions: { [versionId]: Version }`：文件历史版本。`{ versionId, entryId, spaceId, groupId, versionNo, versionFileName, fileSize, fileHash(SHA-256), storageRelativePath, uploadedBy, uploadedAt, changeNote }`。
- `logs: LogEntry[]`：`{ logId, groupId, spaceId, entryId, operatorId, action, detail, createdAt }`，`action` ∈ create_space/create_folder/upload_file/upload_version/download/rename/delete。

物理布局：`files/<目录树>/<文件含全部版本>`、`.trash/<删除项>`。

## 4. 新增 P2P 事件 / 接口

沿用项目「加密单播 + 群系统消息」风格，未强行套用题面英文事件名，而是统一封装：

**控制信令（加密单播请求-响应，`p2p.sendShare`）** — `share` 包：`{ kind:'req'|'res', reqId, action, data }`

| action | 对应题面事件 | 说明 |
| --- | --- | --- |
| `dir_list` | share_dir_list_request/response | 目录列表 |
| `folder_create` | share_folder_create_request | 新建文件夹 |
| `history` | share_file_history_request/response | 历史版本 |
| `rename` | （重命名） | 重命名 |
| `delete` | （删除） | 删除 |
| `download` | share_file_download_request | 下载请求（响应后由主机经 FileTransfer 回传文件） |

**群系统消息广播（复用 `sendRoomStored`，仅聊天框展示）**

| system | 对应题面事件 |
| --- | --- |
| `share-space-created` | share_space_created |
| `share-folder-created` | share_folder_created |
| `share-file-uploaded` | share_file_uploaded |
| `share-file-version-uploaded` | share_file_version_uploaded |
| `share-renamed` / `share-deleted` / `share-space-deleted` | （重命名/删除/删除空间） |

> share_space_host_offline 以「主机在线检测 + 界面离线提示」实现（无需独立包）。share_space_list_request/response 以「成员本地 `shareSpaces` 列表 + 广播增量同步」实现（无需轮询请求）。

**文件传输上下文（`FileTransfer.sendFile` 第 10 参 `share`）**：`{ op:'upload'|'download', spaceId, groupId, parentId, intent('new'|'version'), entryId, reqId, fname, ... }`。

**渲染层 API（`api.share.*`）**：`list / create / deleteSpace / chooseDir / pickFiles / pickFolder / dir / history / createFolder / rename / remove / upload / uploadFolder / download / logs / onChanged / onDownloaded / onDownloadFailed`。

## 5. 版本命名规则说明

实现于 `sharespace.js`：`parseLogicalName()` / `versionFileName()` / `padVersion()`。

- 版本 0（原始）：`需求文档.docx`；版本 k：`需求文档_VKK.docx`，至少两位补零（V01..V99），超过 99 顺延 V100/V101。
- **新版本命名永远基于目标条目的逻辑基名**（`logicalBase`），忽略上传文件原名 → 上传 `需求文档_修改版.docx` 作为新版本，仍生成 `需求文档_V04.docx`。
- 仅当主体以 `_V\d{2,}` 结尾才识别为版本号，因此基名为 `需求文档_V02.docx` 时识别基础名为 `需求文档`，下一版 `需求文档_V03.docx`，**不会出现 `_V01_V02` 叠加**。
- 主列表显示逻辑名（指向最新版本），历史版本列表显示各物理版本名。

## 6. 共享空间存储目录说明

- 默认：`<应用数据目录>/group_shares/{groupId}/{spaceId}/`（`groupId` 中的 `:` 等非法字符做路径净化）。
- 可在「新建共享空间」时自定义本机存储目录。
- 子结构：`files/`（真实文件含历史物理版本）、`.ilink-share/meta.json`（元数据）、`.trash/`（删除项）。
- 仅共享主机本机保存真实文件；成员本地只存空间列表与目录缓存快照。

## 7. 异常处理说明

- **路径穿越**：`checkSegment()` 拒绝 `..`/分隔符/盘符冒号；`safeFileName()` 对上传名取 basename 净化；`isInside()` 对最终落盘绝对路径做越界兜底——三重防线。
- **Windows 非法文件名/保留名**：拒绝 `\ / : * ? " < > |`、控制字符、`CON/PRN/AUX/NUL/COM1-9/LPT1-9`、以点或空格结尾。
- **重名**：新建文件夹重名报错；上传同名文件不静默覆盖——返回 `conflict`，UI 弹窗让用户选「作为新版本（生成 _VXX）/ 改名为新文件 / 跳过」。
- **主机离线**：在线检测后禁止远程上传/下载/新建/查看，明确提示「共享主机离线，暂时不可访问」；有缓存则只读展示并标注「缓存数据，可能不是最新」。
- **传输中断**：复用 FileTransfer——未收 EOF 不落地、保留 `.part` 供续传；落盘前 `.part` 暂存校验后改名，不生成半成品版本。
- **Hash 校验**：上传由主机重算 SHA-256 入库；下载由 FileTransfer 对整文件校验 SHA-256，不一致即删 `.part` 报失败。
- **大文件**：流式分片（64KB），不一次性读入内存（100MB 实测见下）。
- **权限**：非群成员的信令请求拒绝；删除/重命名限主机或群主。
- **日志**：所有操作写入 `meta.json.logs` 与运行日志 `ilink.log`，出错记录原因。

## 9. 尚未实现 / 后续建议

1. **版本回滚**：第一版未做。若实现，按要求将回滚动作也生成新版本（不覆盖）。
2. **超大目录列表**：单次目录列表走 UDP（封顶约 60KB，约 100+ 条目）；超限会提示「分目录查看」。后续可改为 TCP 分页传输。
3. **删除恢复 UI**：已移入 `.trash` 可在主机本地手工恢复；尚无界面化「回收站还原」。
4. **断点续传进度可视化**：共享空间上传/下载复用既有进度事件，但面板内尚未单独展示每文件进度条（仅状态文字），可后续增强。
5. **多机真机联调**：建议用 `npm run dev` + `npm run dev:second` 双实例在同网段按上表 T1–T15 手工回归一次。

## 10. 删除与安全策略说明（按要求选择并说明理由）

- **删除策略**：选择「移入 `.trash` 并标记 `status=deleted`，历史版本一并进回收站、主列表隐藏」，而非物理删除。
  **理由**：纯 P2P 无服务端回收兜底，物理删除不可逆、风险高；移入回收站可恢复、保留审计与历史版本，且主列表已隐藏不影响日常视图，最安全。
- **上传文件夹**：选择「逐文件上传并保留目录结构」（而非压缩传输再解压）。
  **理由**：复用既有稳健的单文件 TCP 传输（加密/校验/续传），实现简单、单点失败影响小、无需引入压缩依赖，符合「第一版优先稳妥」。

## 11. 是否需要手动配置目录、权限或防火墙

- **目录**：无需手动配置。默认存于应用数据目录 `group_shares/`，会自动创建；也可在创建空间时自定义目录。
- **权限**：确保应用对所选存储目录有读写权限（自定义目录请选有写权限的位置）。
- **防火墙**：与现有文件传输一致——共享空间文件走既有 TCP 文件端口、控制信令走既有 UDP 单播端口。**首次运行如系统弹出防火墙提示，请允许本应用在专用/专网通信**；若此前已为 iLink 放行则无需额外配置。无新增对外端口。
