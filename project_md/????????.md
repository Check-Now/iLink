# iLink (freedom) 代码优化分析报告

> 目标：稳定性 / 可维护性 / 可测试性。
> **本报告仅为分析与建议，未改动任何源码、未执行测试或构建。** 所有结论均基于对仓库当前代码的静态审阅，行号对应审阅时的文件状态。
> 审阅范围：`electron/*.js`、`src/App.jsx`、`src/main.jsx`、`project_md/*`、`package.json`、构建配置。

## 0. 总体判断

代码整体质量较高：端到端加密、离线发件箱、断点续传、路径穿越防护、权限"主进程 + 领域层双校验"等关键安全/可靠性设计都在位，注释充分。主要问题集中在**三个超大文件的职责过载**、**大量重复的校验/状态/事件样板**、**少数静默失败与资源未释放的稳定性缺口**，以及**完全没有测试基建**（与文档描述不符）。

## 2. 关键问题与建议改法

每条含：原问题 / 建议改法 / 为什么这样 / 对功能影响 / 协议·存储·兼容性。

### D1 · Vault 解密失败导致静默数据丢失（高）

**原问题**：`unlock()` 中
```js
try { data = JSON.parse(cryptoMod.decryptBuf(key, fs.readFileSync(this.storePath)).toString('utf8')) }
catch (_) { data = {} }
...
this._saveNow()  // 立即把空 data 写回 store.enc
```
当 `verifier` 校验通过（密码正确）但 `store.enc` 损坏/截断/被外部进程写坏时，`data` 被置空，`this.data = _ensureFields({})` 生成全新身份与空历史，随后 `_saveNow()` **用空数据覆盖了那个尚可能可抢救的密文**。结果：联系人、群组、聊天记录、发件箱、置顶、共享空间全部永久丢失，且不可逆。

**建议改法**：区分"账户不存在/密码错误"与"store 解密失败"。解密失败时：① 不覆盖，先把损坏文件改名备份（`store.enc.corrupt-<ts>`）；② 抛出可识别错误交由 UI 提示"数据可能损坏，已生成备份"，或在用户明确确认后才以空库继续。

**为什么**：宁可让用户看到错误并保留备份，也不能静默清空唯一的本地数据副本。这是数据完整性的底线。

**影响**：仅改变异常路径，正常解锁不受影响。**涉及存储**，不涉及网络协议；旧数据完全兼容（只是不再被覆盖）。

### D2 · P2P 单播发送缺 socket 空指针防护（中）

**原问题**：`reconnect()` 期间 `this.uniSock = null`（`p2p.js:176-179`）。而 `sendTyping`(577)、`sendNudge`(384)、`sendRoomAvatar`(593)、`sendTypingRoom`(601)、`_signal`(614-623) 直接 `this.uniSock.send(...)`，仅判断 `peer.online`，未判断 socket 存在。若 IPC 在重连窗口触发（如此时用户正输入触发 typing），抛 `TypeError` 未捕获。`_sendAck`/`resendPrivate`/`sendShare` 已有 try/catch 或前置判断，行为不一致。

**建议改法**：统一一个 `_unicast(peer, buf)`：内部判 `this.uniSock && peer.address && peer.uport`，`try/catch` 吞瞬时错误。所有单播改走它。

**为什么**：消除散落的空指针面，并把"瞬时网络错误可忽略"的既定策略集中一处。

**影响**：纯防御性，无行为变化。不涉及协议/存储。

### D3 · 文件发送端无空闲超时，异常对端致挂起 + 发件箱永久卡 mid（中）

**原问题**：`sendFile` 在 `socket.on('connect')` 里 `socket.setTimeout(0)`（`filetransfer.js:136`）彻底取消空闲超时。之后若接收端进程被强杀/网络分区（无 FIN/RST），发送端 `socket.write` 因背压 `drain` 永不到来 → 读流 `pause` 后永挂，既不 `sent` 也不 `failed`。而 `main.js` 的 `inFlight`（703）依赖这两个事件来 `delete(mid)`；`outboxDrain`（725）对 `inFlight.has(mid)` 的条目 `continue` 跳过。于是该文件**再也不会被补发**，直到进程重启。

**建议改法**：保留一个较宽松的整体/空闲超时（如基于"两次分块写入间隔"重置的 `setTimeout`），超时 `socket.destroy()` 并 `emit('failed')`；`main` 侧也可加兜底：`stopP2P`/重连时清空 `inFlight`，或对长时间无进度的 mid 主动失败。

**为什么**：可靠投递的前提是每个传输最终都收敛到 sent 或 failed，不能有"既不成功也不失败"的黑洞。

**影响**：仅影响异常路径；正常传输无变化。断点续传仍有效（失败保留 `.part`）。不涉及协议字段。

### D4 · finalizeFile group 分支先落盘再 return（低，且不可达）

**原问题**：`main.js:679-682`
```js
const dest = uniquePath(path.join(dir, safeName))
moveFile(info.tempPath, dest)            // 先移动到下载目录
if (info.scope === 'group') return       // group 时直接返回，不入库/不通知
```
group scope 文件被搬进下载目录却无任何会话记录或提示（"幽灵文件"）。同时实际发送路径只产生 `private`/`room`/`share`，从不产生 `group` scope 文件，故此分支不可达。

**建议改法**：移除该死分支；若确需保留防御，则把判断提前到 `moveFile` 之前并记录日志。

**影响**：消除误导逻辑，无功能变化。

### D6 · 组件卸载后 setState（低）

**原问题**：`showChatNotice`(2188)、onNudge 的 `setShake`(2247)、`removeToastSoft`/`scheduleToastExpiry`(2410-2413) 用裸 `setTimeout` 改 state；锁定时 `App` 卸载整个 `ChatScreen`，这些定时器仍会触发。React 18 不再告警，但属未清理资源。

**建议改法**：用统一的"已挂载"ref 守卫，或把这些定时器登记到一个 `timersRef` 并在卸载 `useEffect(()=>()=>clearAll(),[])` 清理（`nudgeHoverTimerRef` 已是范例，`App.jsx:2871`）。

**影响**：纯清理，无行为变化。

### R1 · IPC 解锁守卫重复 79 处（可维护性，高收益）

**原问题**：几乎每个 `ipcMain.handle` 开头都是 `if (!vault || !vault.unlocked) return ...`，文案与返回结构各异（有的返回 `{}`、有的 `[]`、有的 `{ok:false,error:'应用尚未就绪'}`），既重复又不一致。

**建议改法**：提供包装器，例如
```js
const handleUnlocked = (channel, fn) =>
  ipcMain.handle(channel, (e, ...args) =>
    (vault && vault.unlocked) ? fn(e, ...args) : NOT_READY)
```
并配 `requireGroup`/`requireGroupOwner` 等领域校验（见 R2）。**注意**：必须保持在主进程/领域层校验，不能下沉到前端（满足约束第 6 条）。

**为什么**：一处定义"就绪/未就绪"的语义与返回，消除 79 份漂移。

**影响**：行为等价（统一返回需逐一核对各 handler 既有返回形态，避免破坏前端判空）。不涉及协议/存储。

### R6 · `safeFileName` 两份且行为不一致（一致性 / 安全）

**原问题**：`pathutil.safeFileName`（聊天文件落地用，`main.js:679` 调用）不拒绝 Windows 保留名（CON/PRN/...）也不去结尾点/空格；`sharespace.safeFileName`（群空间用）两者都处理。同一"对端可控文件名净化"在两条路径产生不同结果——对端发名为 `CON` 或 `report.` 的聊天文件可能落到下载目录形成异常文件名。

**建议改法**：统一到一份较严格实现（含保留名 + 结尾点/空格处理），两处共用。`pathutil` 已是纯函数模块，适合作为单一真源。

**影响**：聊天文件落地命名更稳健；不改变协议（文件名仍来自元数据），属本地落盘行为收敛。

### R5 · 头像可广播化三份实现

**原问题**：`avatarutil.publicAvatar`（纯函数，限长 + 静态帧兜底）、`main.publishableAvatar`（多一层 nativeImage 缩略图兜底）、`main.groupAvatarPayload`（群头像，逻辑与 publishableAvatar 高度重叠）。三者关于"原图 → 静态帧 → 缩略图"的降级策略各写一遍。

**建议改法**：以 `avatarutil` 为核心降级策略，`main` 仅在其返回需要时追加 nativeImage 缩略图兜底（已有注释说明 p2p 层不能依赖 electron），`groupAvatarPayload` 改为调用同一核心。

**影响**：行为应保持一致（需对照三者的尺寸阈值 `AVATAR_MAX_CHARS`/JPEG 质量逐一核验）。**头像随 presence/room-avatar 包下发，属协议载荷**——合并时务必保持下发字节与现状等价，避免老客户端解析差异。

### A1 · main.js 拆分（可读性 / 可测试性）

**原问题**：单文件承担窗口、托盘、通知、P2P 事件编排、离线发件箱、文件落地、**群共享空间宿主编排**（`shareHosts`/`shareRequest`/`onShareSignal`/`onShareFileDone`/`notifyShareSync`，约 839-1170 + 1736-2037）、**置顶消息**（快照/校验/同步，约 1479-1620）、截图、贴纸、全部 IPC。

**建议改法**（保持现架构，仅做模块边界整理）：
- `electron/sharespace-host.js`：群空间宿主编排与信令处理（依赖注入 `vault`/`p2p`/`ft`/`sendToRenderer`）。
- `electron/pinned.js`：置顶快照构建、`pinnedMessageTypeOf`/`buildPinnedSnapshot`/同步信令。
- `electron/outbox.js`：`outboxDrain`/`outboxDrainAll`/`inFlight` 状态机。
- `electron/windows.js` + `tray.js`：窗口与托盘/通知/图标生成。
- `main.js` 收敛为：生命周期 + IPC 注册（每个 handler 只做校验并委托领域层）。

**为什么**：缩小单文件认知负担，使发件箱状态机、置顶、群空间可被单测（当前它们与 electron/IPC 强耦合，无法独立测）。

**影响**：纯结构性重构，**必须配测试护栏**（见第 3 节）并逐模块小步迁移，避免一次性大改引入回归。不涉及协议/存储。

### A6 · 存储内联预览导致全量重写（性能 / 可扩展）

**原问题**：图片消息把 `dataUrl`（内联预览，最高 2MB，见 `PREVIEW_INLINE_MAX`）存进 history（`main.js:667-671,1199`）。`_saveNow` 每次把**整个 data 对象**序列化 + AES-GCM 加密 + 写盘（300ms 去抖，`vault.js:218-229`）。随会话/图片增多，单次保存成本与 `store.enc` 体积线性膨胀，可能造成主线程卡顿与磁盘占用过大。

**建议改法**：缩略图与正文分离——预览写独立缓存文件（或仅保留磁盘原图路径，渲染层按需生成缩略图），history 仅存 `path` 与小尺寸占位。或显著下调内联阈值。

**影响**：**涉及存储结构**，须走 `Vault._ensureFields` 迁移：旧数据里已内联的 `dataUrl` 需兼容读取（保留旧字段读取路径，新写入走分离方案）。属可选优化，建议单独一轮并充分回归。

## 3. 测试与回归建议

### 现状

**零测试**：无 `test/` 目录、devDependencies 无任何测试运行器、`npm test` 为 `echo "no tests configured" && exit 0`。讽刺的是 `pathutil.js`/`avatarutil.js`/`sharespace.js`/`crypto.js` 都刻意写成 electron-free 并注明"便于单测"，但因无基建，这份可测性红利完全未兑现。文档（`PROJECT_CONTEXT.md`/`CODE_INDEX.md`）仍描述存在 `test/ack-smoke.js`，与现状不符。

### 建议（零新增重型依赖）

用 Node 内置 `node:test` + `node:assert`（无需新依赖，契合约束第 3 条），把 `npm test` 指向 `node --test test/`。最小高价值用例：

1. **crypto**：`encrypt/decrypt`、`encryptBuf/decryptBuf` round-trip；`deriveKey` 双方对称一致；篡改 tag 应解密失败。

## 4. 风险与后续建议

### 4.1 本轮已识别但未深入的点

- **D3 的发送端超时**与 `inFlight` 黑洞需要在真实弱网下验证触发条件后再定方案，避免误伤慢速大文件传输。
- **A6 存储分离**改动面大且涉及迁移，建议独立成轮。
- `settings:set` 未对 `patch` 做键白名单，渲染层理论上可写入任意键到持久化 settings（`vault.setSettings` 直接展开合并，`vault.js:663`）。本地可信场景风险低，但建议加白名单做纵深防御。
- `p2p` 的 `seen` 去重缓冲（`SEEN_MAX=800`）与 share/pin 复合 key 共用同一缓冲，极端高频下可能加速淘汰消息 mid；当前规模无碍，规模上升后需关注。
- `presence` 事件每心跳（~2s）触发 `outboxDrain`+置顶拉取，置顶已节流（10s），drain 幂等但偏频繁；可加最小间隔节流。

### 4.2 建议的下一轮优先级

### 4.3 建议遵守的约束复述
