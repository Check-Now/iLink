## 第二步～第三步：反向识别到的功能模块总览

按代码实际发现（非常见功能套模板）：

1. **认证与本地加密存储**：首次设密、解锁、改密、锁定、重置身份；scrypt KDF + AES-256-GCM 整库加密。
2. **端到端加密原语**：X25519 身份密钥对、ECDH+HKDF-SHA256 会话密钥、AES-256-GCM 消息/文件加密。
3. **局域网发现与在线状态**：UDP 广播 presence + 心跳（2s）+ 离线判定（6s）+ 首见单播回包 + 网卡变化/断线自愈重连。
4. **私聊**：UDP 单播 + ACK 确认 + 超时重发（3 次）+ 按 mid 去重 + 持久化离线发件箱（queued→上线补发）。
5. **群聊**：逐成员公钥加密（非共享群密钥）、在线广播 best-effort、离线成员可靠单播补发（did 跟踪）。
6. **文件传输**：TCP 直连、64KB 分块独立加密、`.part` 断点续传、SHA-256+大小校验、取消、超时、过期 part 清理。
7. **消息记录**：加密持久化、迟到消息按时间序插入、单会话 1000 条上限、保留天数裁剪、未读位（reads）跨重启恢复。
8. **撤回 / 表情回应 / 正在输入 / 拍一拍**：均有协议包与本地落库。
9. **群管理**：建群、加成员、移成员、退群、解散、转让群主、群头像同步（逐成员加密单播）。
10. **群消息置顶**：群内置顶（上限 10）、置顶操作日志、跨成员 P2P 同步拉取/合并。
11. **群共享空间**：宿主权威存储 + 成员缓存快照；文件夹/上传/下载/重命名/删除/搜索；路径穿越与保留名安全校验。
12. **截图**：desktopCapturer 冻屏 + 选区窗口 + 复制/保存/发送。
13. **头像**：选图、裁剪（缩放/定位）、文字头像、GIF 动图（超限退静态首帧）。
14. **表情包（贴纸）**：导入/列出/删除，按贴纸标记发送。
15. **粘贴图片发送**：dataURL 落地临时 PNG 进待发区。
16. **设置**：主题/UI 风格（11 种）/字体/通知开关+预览/正在输入开关/发送键/最小化到托盘/开机自启/关闭行为/接收模式（自动/手动）/下载目录/单文件大小上限/Markdown/免打扰/在线状态/状态签名/UDP 端口/广播地址。
17. **系统集成**：托盘（在线/未读角标/免打扰图标/闪烁）、系统通知、任务栏闪烁、未读角标、独立聊天窗口、危险扩展名打开确认、打开文件所在目录、外链用默认浏览器打开、打开日志目录。
18. **联系人**：在线节点与本地联系人合并、本机备注（不外发）、黑名单字段（见风险）、静音/置顶会话、最近会话元数据。
19. **匿名模式 / 阅后即焚**：presence 显示「匿名」；burn 消息不落库、不入发件箱、不补发。
20. **检查更新**：仅占位（见风险）。

## 第五步：“假功能”专项核查结论

逐项核对题目所列 12 类“假功能”嫌疑：

| 嫌疑点 | 核查结论 |
| -- | -- |
| 按钮点击无逻辑 | 未发现。`api.*` 调用点覆盖全部 preload 方法，按钮均接真实 IPC。 |
| 页面没接真实接口 | 未发现。各页面初始化拉取真实数据并订阅真实事件。 |
| 前端显示成功但无后端 ACK（私聊） | 不成立。私聊有真实 UDP ACK + 超时重发 + delivered 状态（动态验证）。 |
| 进度条非真实进度 | 不成立。进度条宽度由 `file:progress` 的 `received/size` 真实驱动。 |
| 在线状态无心跳 | 不成立。2s 心跳 + 6s 离线 sweep + 网卡变化检测。 |
| 聊天记录无持久化 | 不成立。AES-GCM 加密落库，动态验证可跨重启恢复。 |
| 文件发送入口无真实上传/下载 | 不成立。完整 TCP 端到端，动态验证字节一致。 |
| 断点续传按钮无 chunk 状态 | 不成立。`.part` + resumeFrom 偏移协商真实存在（逻辑已验证）。 |
| 设置未真正保存 | 不成立（除 blacklist 外）。`vault.setSettings` 持久化。 |
| 管理页面无权限校验 | 部分成立。移成员/解散/改群头像有群主校验；**转让群主缺 IPC 层校验**（问题 #5）。 |
| 消息成功仅前端状态变化 | 不成立（私聊/群离线补发有 ACK；群在线广播为 best-effort，见问题 #4）。 |
| 文件完成仅 UI 变化无 hash 校验 | 不成立。收尾重算 SHA-256 + 大小校验，失败删 .part 不落盘。 |

**真正需要标注的非完整项**：拉黑（未实现/被禁用）、检查更新（仅占位）、转让群主（权限风险）、群在线消息可靠性（best-effort 风险）。

## 第十步：最终结论

**1. 已完整实现（可实际使用，且核心逻辑动态验证通过）**
本地加密身份与解锁、端到端加密、局域网发现与在线状态、私聊（含 ACK/重发/去重/离线补发）、文件传输（分块加密/SHA-256/大小校验/取消/手动接收/断点续传逻辑）、消息持久化与时间序/未读恢复、撤回/表情/正在输入/拍一拍、群管理（建群/加移成员/退群/解散/群头像）、群消息置顶、群共享空间（含路径安全）、阅后即焚/匿名、设置持久化、日志审计。

**2. 仅部分实现**
群聊在线送达（离线补发可靠，在线广播为 best-effort）。

**3. 仅 UI / 未实现**
拉黑/黑名单（字段在、逻辑被禁用且设置被强制清空）；检查更新（占位，无真实更新源）。

**4. 存在明显风险**
群在线消息可能丢失（P1）；转让群主越权（P2）；黑名单失效（P2）；presence dnd 不一致（P3）。

**5. 无法判断（需运行环境）**
所有 GUI 渲染与系统集成：截图实拍、托盘/通知/角标/独立窗口/开机自启的真实表现；跨真实子网的多机发现、广播地址、断网重连的现场行为。原因：沙箱为 Linux 无显示环境且仅单机，无法启动 Electron GUI 与多机局域网。

**6. 核心可用程度**
**高。** 这是一个完成度很高、工程质量良好的真实产品，绝非“假界面”。加密、存储、收发、文件传输等关键链路均为完整实现并通过动态自测。

**7. 最需优先修复**
P1 群在线消息可靠性 → 其次 P2 转让群主越权与黑名单失效。

**8. 下一步建议补齐**
群在线 ACK 补发；群主转让权限校验；黑名单接通或下线；presence dnd 白名单；（可选）检查更新接入。

**9. 暂不建议开发**
中心化服务器/账号体系（与无服务器 P2P 定位冲突）；复杂的前向保密握手（内网场景收益有限，可后置）；自动更新（绿色版可暂缓）。

**10. 发布内网测试版的最低补齐项**
- 必须：修复 **P1 群在线消息可靠性**（避免群里丢消息这一硬伤）。
- 必须：补 **转让群主权限校验**（防越权）。
- 必须：对 **黑名单** 做决断——接通或从设置 UI 隐藏，避免给用户“已拉黑”的错误安全感。
- 建议：**检查更新** 改文案或接入更新源，避免误导。
- 建议：在 ≥2 台真实 Windows 机器上完成一轮 GUI + 多机发现/收发/文件/群聊回归（覆盖本报告“无法判断”项）。
