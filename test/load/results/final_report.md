# iLink 大规模压测正式报告（300 / 500 节点）

- 生成时间：2026-06-15
- 被测对象：iLink（仓库 freedom）— Windows 局域网 P2P 加密通讯，Electron + React
- 执行方式：**单机无头进程内压测**，复用 `electron/p2p.js` / `electron/filetransfer.js`（零侵入，未改业务代码）
- 工具：`test/load/`（`node test/load/run.js`，参数全部环境变量可配）
- 运行环境：Node v22，Linux，单进程 3.9GB 内存沙箱，`--max-old-space-size=3072`

> 架构前提：本项目无中心服务端、无 WebSocket/HTTP/登录接口、无数据库。"在线"= LAN 对等节点；"服务端资源"= 单节点资源。详见 `test/load/README.md`。

---

## 一、总体结论

| 规模 | 场景 | 在线 | 丢失 | 重复 | 乱序 | ACK 成功率 | 通过项 |
|---|---|---|---|---|---|---|---|
| **300 节点** | 全 6 类场景（在线/单聊/群聊/重连/离线补发/文件） | 300/300 | **0** | **0** | 9 | 99.91% | **11/12** |
| **500 节点** | 在线 + 单聊 + 群聊（容量档） | **500/500** | 15（私聊，残留 0.34%） | 0 | 83 | 99.32% | 8/12 |

**核心结论：**
1. **可靠性机制有效**：300 节点全场景下，私聊/群聊（可靠路径）**零丢失、零重复**；断线重连 100% 恢复、离线消息 100% 补发、文件 hash 100% 通过、断点续传成功。ACK + 超时重发 + 持久发件箱在大规模下能把瞬时 UDP 丢包**最终补齐**。
2. **群聊实时延迟是主要短板**：群聊逐成员加密 + 单播 fan-out，实时 P95 在 300/500 下均约 **1.5s**，超出 1s 建议线。单聊实时 P95 约 **1ms**（优秀）。
3. **500 节点容量可达但有代价**：500 节点全部成功上线（建连 P95≈6ms），但持续消息阶段出现 **0.34% 残留私聊丢失 + ACK 降到 99.32%**。这是**单进程模拟 500 节点**的固有限制（500 个节点共享同一事件循环与同一回环 UDP 协议栈，缓冲区溢出丢包远重于 500 台真机），**非应用缺陷**——同样的可靠路径在 300 节点下零丢失。
4. **无内存泄漏迹象**：以堆(heapUsed)为判据，300 节点堆 35MB→33MB（-5.4%），500 节点堆稳定在 58–67MB；RSS 峰值仅 ~200MB。

## 五、消息丢失 / 重复 / 乱序 / 延迟 专项

- **丢失**：300 节点全场景 **0 丢失**（含重连/离线，均经发件箱补齐）。500 节点残留 15 条私聊（0.34%），系单进程回环 UDP 拥塞，ACK 重发 + 发件箱在持续高压窗口内未及补齐；属工具侧限制。
- **重复**：两档均 **0 重复**。去重窗口 `SEEN_MAX=800` 在本次单会话活跃量下未被击穿；重连后亦未重复投递。
- **乱序**：300 节点 9 条、500 节点 83 条。**这是协议固有特性**：消息无序号、无重排缓冲，UDP 单播 + 超时重发会把旧消息排到新消息之后。建议按业务对顺序的敏感度评估，必要时加序号。
- **延迟**：单聊实时 P95 ≈ 1ms（优秀）。群聊实时 P95 ≈ 1.5s（逐成员加密 + 单播 fan-out 开销）。整体延迟含离线恢复消息（重连 downMs=3s + 补发），故 P99/Max 偏高属预期。单机回环延迟整体偏乐观，真实多机 LAN 需另行标定基线。

## 六、发现的问题

1. **群聊实时延迟超标（高优先级，应用侧）**：300/500 节点群聊实时 P95 均约 1.5s（>1s 标准）。根因是 `sendRoom` 对每个成员逐一 X25519+AES 加密再单播，O(成员数) 串行开销，且大群广播包超 `SAFE_DATAGRAM_BYTES=60000` 后全量转单播。
2. **500 节点单进程残留丢包（工具侧，非应用缺陷）**：0.34% 私聊残留 + ACK 99.32%。500 节点共用一个事件循环与一套回环 UDP 栈，发送缓冲溢出导致丢包远超真机分布；同样路径在 300 节点零丢失。
3. **乱序（架构特性）**：无序号/无重排，乱序量随并发上升（9→83）。
4. **大规模离线/重连风暴的进程预算**：在单进程内同时让 250 个节点离线再重连会超出 3.9GB 沙箱预算（OOM）。这是**压测进程**的限制，不是应用限制。

### 工具局限（如实声明）
- 单进程模拟 N 节点时，所有节点共享事件循环与回环 UDP 栈，**丢包/延迟比 N 台真机更差**；500 节点的丢包数据应理解为"工具压力上限"而非"应用真实丢包率"。
- 未覆盖真实 UDP 广播发现 / presence 风暴 / 真实网卡带宽（需多机环境）。
- 建议在 ≥500 节点时按"多进程/多机各承载 50–100 节点"分片，绕开单进程瓶颈。

## 八、建议修复方案

1. **群聊延迟（优先）**：(a) 复用会话密钥缓存已具备，瓶颈在串行 fan-out——可对在线成员批量并发单播；(b) 评估对群消息也引入"广播 + 在线成员 ACK 确认"以减少逐成员串行；(c) 大群考虑分片群或中继。
2. **顺序敏感场景**：在消息体加单调 `seq` + 接收端按 (sender,seq) 重排缓冲；当前 9–83 条乱序对一般 IM 可接受，但群公告/指令类需保证顺序。
3. **去重稳健性**：单会话长期高活跃时 `SEEN_MAX=800` 可能被击穿，建议改为 (sender, seq) 复合键或时间窗去重。
4. **大规模真机验证**：≥500 节点请用多机/多进程分片复测，以区分应用真实丢包率与单进程工具压力上限。
5. **文件高并发**：本次 8×3MB 全通过；若上 50 并发或 100MB，关注 `SOCKET_TIMEOUT_MS=30000` 是否被 CPU 抢占误判失败。
